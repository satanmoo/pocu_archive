## `xlat`: 테이블 룩업

![[Pasted image 20260314005718.png]]

"translate"의 약자이다.

> [!note] `xlat` 동작
> `AL = DS:[BX + AL]` — BX(테이블 시작 주소) + AL(색인)로 1바이트를 읽어 AL에 저장한다.
>
> ```masm
> lea bx, table
> mov al, 02h
> xlat              ; AL = table[2]
> ```
>
> `xlat table2`처럼 피연산자에 테이블 변수를 직접 쓸 수도 있다.

```asm
TITLE Main

  

.DOSSEG

.8086

.NO87

.MODEL TINY

  

.DATA

table DB 01, 02, 03, 04, 05

table2 DB 10h, 20h, 30h, 40h, 50h

  

.CODE

.STARTUP

lea bx, table

mov al, 02h

xlat

xlat table2

exit:

; 프로그램 종료

mov ah, 4Ch

xor al, al ; 리턴값 0

int 21h

END
```

위 코드를 어셈블 해보자

![[Pasted image 20260314011029.png]]

`lea bx, table` 은 다음과 같이 **Code View**에서 보여준다.
- `LEA BX,WORD PTR [010E]`
- 대괄호가 있어서 `010E` 주소에서 무엇을 할 것 처럼 보이지만, `LEA`는 주소를 로드할 뿐 **절대 역참조 하지 않는다.**

![[Pasted image 20260314011533.png]]

`010E` 주소에 덤프해보면 "01 02 03 04 05 10 20..." `table`의 주소가 맞다.

![[Pasted image 20260314011637.png]]

살행하고 `BX`의 값을 확인해보니 `table`의 주소가 로드된 것을 확인할 수 있다.

> [!Tip] `LEA` vs `MOV OFFSET`
> 
> 위 동작은 `MOV BX, OFFSET table`과 동일하다. 둘 다 BX에 주소 자체를 넣는다.
> 
> 사실 단순 변수 주소는 `MOV BX, OFFSET table`처럼 `MOV`와 `OFFSET`의 조합으로 충분하다.
>
> LEA의 진짜 용도는 `lea bx, [bp+si+4]`처럼 복합 주소 계산이 필요할 때다. OFFSET은 어셈블 시점 상수만 처리하지만, LEA는 실행 시점에 레지스터 조합으로 주소를 계산할 수 있다.

![[Pasted image 20260314012035.png]]

`mov al 02h`를 실행했다. 

`AX`의 값이 `0002`로 바뀐 것을 확인할 수 있다.

![[Pasted image 20260314012306.png]]

`xlat`을 실행했다.

`AL`의 값이 `03`으로 바뀌었다.

최종주소(`table` 주소 + 색인값 2 )에 해당하는 값이 `03`이기 때문이다.

![[Pasted image 20260314012435.png]]

한 번 더 `xlat`을 실행했다.

`AX`값이 04가 되었다.

최종주소는 `table` 주소 + 색인값 3으로 계산되었다.

피연산자로 넣은 `table2`는 의미가 없다.

어셈블 결과를 확인해봐도 `xlat`이나 `xlat table2`나 동일한 것을 확인할 수 있다.

> [!note] `xlat` 피연산자의 정체
> `xlat table2`의 피연산자는 테이블 전환이 아니라 세그먼트 오버라이드 용도다. MASM이 해당 변수의 세그먼트를 확인해서 기본 DS 대신 다른 세그먼트 프리픽스를 붙여준다.
>
> TINY 모델에서는 모든 세그먼트가 동일하므로 의미 없다. 디스어셈블리에서 `xlat`과 `xlat table2` 모두 `D7` 1바이트로 인코딩된 것이 그 증거.
> - `table` 과 `table2`의 세그먼트는 동일하다.

따라서 의도대로 `table2[3]`의 값을 찾기 위해서는 `xlat table2` 전에 `lea bx, table2`로 `bx`값을 수정해야 한다.
