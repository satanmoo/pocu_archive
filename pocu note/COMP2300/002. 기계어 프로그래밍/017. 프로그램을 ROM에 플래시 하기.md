### bin 파일로 저장하기

![[Pasted image 20251228200109.png]]

바이너리 파일을 뽑아내는 프로그램을 작성하자. 목적이 바이너리 파일을 뽑아내는 것이기 때문에 어떤 고수준 언어로 작성해도 상관없다.

바이너리 파일의 이름은 `"program.bin"`이다.

`bin_out` 은 버퍼의 변수명이다.
- `0`으로 초기화한다.

`SIZE (0x8000)`은 바이트 버퍼의 크기다.
- **2^15**B
- 32KB

ROM에 들어갈 모든 내용을 메모리에 한 번에 저장하려는 의도이다.

`char`는 8비트 가정

![[Pasted image 20251228200716.png]]

`0xA9, 0xFF, 0x4C, 0x00, 0x80`은 위에서 초록색으로 하이라이트한 기계어다.

위에서 선언한 `bin_out`에  `memcpy`로 복사한다.

`bin_out` 의 시작부터 `0xA9, 0xFF, 0x4C, 0x00, 0x80` 내용이 복사되고, 나머진 0으로 남아있다.
- ROM에서 프로그램의 시작이 "program.bin"파일의 첫번째 값부터 시작한다.

"program.bin"파일의 첫번째 값은 ROM 입장에서 **0x0000**주소에 매핑된다.
- `bin_out` 배열의 0번 인덱스가 ROM의 **0x0000**주소에 해당한다.

CPU 입장에서는 주소를 절반만 사용한다. ([[014. ROM 주소 매핑]], [[015. ROM 설치하기]] 내용 참고)
- 주소 핀 16개 중에서 15개만 사용한다.
- **0x8000** 즉 **1000_0000_0000_0000**이 CPU입장에서 ROM의 첫번째 주소이다.
	- 최상위 비트가 1이다.
	- CPU의 A15 핀에서 1이 나가고 ROM의 CEB에 0이 들어온다.
	- 최상위 비트 1을 제외하면 **0x0000**이다.

```c
bin_out[0x7FFC] = 0x00;
bin_out[0x7FFD] = 0x80;
```
**0x8000**이라는 프로그램의 시작 주소를 ROM의 **FFFC**, **FFFD**에 넣어야 한다.
- 근데 왜 배열에는 **7FFC**와 **7FFD**일까?
	-  ROM은 애초에 주소 핀이 15개이기 때문에 32KB만 사용한다. 따라서 CPU 입장에서 **FFFC**, **FFFD** 라고 읽어도, ROM의 입장에서 최상위 비트 1을 제외하면  **7FFC**, **7FFD**이다.
- 애초에 `bin_out`의 크기를 32KB로 선언한 것도 이를 ROM이 주소 핀을 15개 사용하는 것을 반영한 것이다.
