# 변수 및 초기화

![[Pasted image 20260308010517.png]]

변수는 메모리 공간을 차지한다.

그림으로 크기를 표현했다.

> [!note] 변수 선언 디렉티브 — Define XXX
> MASM에서 메모리 공간을 예약하고 초기화하는 디렉티브들.
> 접두사 **D = Define**, 접미사는 크기를 나타낸다.
>
> | 디렉티브 | 이름 | 크기 |
> |---|---|---|
> | `DB` | Define **B**yte | 1 바이트 (8비트) |
> | `DW` | Define **W**ord | 2 바이트 (16비트) |
> | `DD` | Define **D**oubleword | 4 바이트 (32비트) |
> | `DQ` | Define **Q**uadword | 8 바이트 (64비트) |
> | `DF` | Define **F**arword | 6 바이트 — 실용적 사용 거의 없음 |
> | `DT` | Define **T**en-byte | 10 바이트 (80비트) |
>
> > [!tip] Word = 16비트인 이유
> > 8086/8088 설계 당시 기본 처리 단위가 16비트였다.
> > "Word"의 크기는 아키텍처마다 다르다 — x86에서 word는 **항상 16비트**로 고정되어 있다.
> > (x86-64로 와도 역호환을 위해 word = 16비트를 유지한다)
>
> > [!note] DT (80비트)는 왜 존재하는가?
> > x87 FPU(부동소수점 코프로세서)의 **내부 연산 포맷**이 80비트이기 때문이다.
> > - 32비트 float, 64비트 double로 최종 저장하기 전에 **정밀도 손실을 최소화**하기 위해 내부적으로 80비트로 계산한다
> > - 최종 결과는 IEEE 754 기준 32비트(float) 또는 64비트(double)로 변환
> > - 일반 어셈블리 코드에서 직접 쓸 일은 거의 없다

## 변수 및 초기화 예시

![[Pasted image 20260308011334.png]]

> [!note] 변수 선언 예시
> ```asm
> .DATA
> byte    DB  23          ; 10진수 23
> short0  DW  12*53       ; MASM이 컴파일 타임에 계산 → 636 저장
> short1  DW  1234h       ; 16진수 즉시값
> int     DD  2147483647  ; 32비트 최대값 (int_max)
> long    DQ  74294967295 ; 64비트값
> empty   DQ  ?           ; 초기화 안 함 → 가비지값, 메모리 공간만 예약
> ```
>
> > [!tip] `12*53` — 컴파일 타임 상수 계산
> > 런타임에 계산하는 게 아니라 **MASM 어셈블러**가 조합 단계에서 계산해 결과값(636)을 직접 기계어에 넣는다.
> > C의 `#define` 매크로 상수와 동일한 개념.
>
> > [!warning] `?` — 초기화되지 않은 변수
> > 메모리 공간만 예약하고 어떤 값도 쓰지 않는다.
> > 실제 실행 시 **그 자리에 남아있던 이전 데이터(가비지)** 가 읽힌다.
> > C의 `int x;` (초기화 없는 지역변수)와 동일한 동작.

> [!note] `.ALIGN` — 메모리 정렬 디렉티브
> ```asm
> num0    DW  1234h
> .ALIGN  4
> num1    DB  12h        ; num1의 시작 주소가 반드시 4의 배수
> ```
> `num0`이 2바이트를 차지한 뒤, `num1`을 4바이트 경계에 맞추기 위해
> **패딩(빈 공간)이 자동 삽입**될 수 있다.
>
> ```
> 주소: [00] [01] [02] [03] [04] ...
>        num0(2B)  pad  pad  num1
> ```
>
> > [!tip] 왜 정렬이 필요한가?
> > CPU는 정렬된 주소에서 데이터를 읽을 때 **1회 메모리 접근**으로 끝낸다.
> > 정렬이 안 맞으면 **2회 접근 후 조합**해야 해서 느려진다.
> > 현대 컴파일러가 구조체 멤버에 패딩을 자동 삽입하는 이유가 바로 이것.

> [!note] `.EVEN` — 짝수 주소 정렬
> ```asm
> num2    DW  1234h
> .EVEN
> num3    DB  12h        ; num3의 시작 주소가 반드시 짝수
> ```
> `.ALIGN 2`와 동일하다. 2바이트(word) 경계 정렬.
>
> > [!tip] 8086 vs 8088에서 `.EVEN`의 의미
> > | | 데이터 버스 | 홀수 주소 접근 |
> > |---|---|---|
> > | **8088** | 8비트 | 1회 = 1바이트씩 읽으니 차이 없음 |
> > | **8086** | 16비트 | 홀수 주소면 2회로 나눠 읽어야 함 → 느림 |
> >
> > 8086 타겟 코드에서 `.EVEN`은 성능에 직접적인 영향을 준다.

## 변수 및 초기화 - 실수

![[Pasted image 20260308012307.png]]

> [!note] 변수 선언 — 실수 (Floating Point)
> ```asm
> .DATA
> float     DD  98.6
> double    DQ  9.312E-7
> edouble   DQ  3EAF3EF274A03CABr  ; 9.312E-7의 IEEE 754 비트패턴을 직접 지정
> ten_byte  DT  7.341E9
> ```
>
> > [!tip] 핵심 개념 — 공간 확보와 값 해석은 별개
> > `DD`, `DQ`, `DT`는 **바이트 공간만 확보**한다. 정수든 실수든 물리적 메모리 크기는 동일하다.
> > 그 공간에 **어떤 비트패턴을 넣느냐**가 의미를 결정하며,
> > 실수 리터럴을 적으면 MASM이 **IEEE 754 비트패턴으로 변환**해서 삽입해준다.
>
> > [!note] `r` 접미사 — IEEE 754 hex 리터럴 (Real Number 로 부른다.)
> > `3EAF3EF274A03CABr`처럼 `r`을 붙이면 **이미 IEEE 754로 변환된 16진수 비트패턴을 그대로** 삽입한다.
> > `edouble`과 `double`은 값은 같지만 표현 방식만 다르다.
>
> > [!warning] Microsoft 자체 실수 포맷 (`r` 없는 hex)
> > MASM 6.1 이전에는 Microsoft 독자 실수 포맷이 존재했으나,
> > **MASM 6.1부터 IEEE 754로 통일**되었다. 현재는 쓸 일이 없다.

이렇게 실수 변수를 작성해도 8088 CPU에서 연산을 할 수 없다.

연산하기 위해서는 8087 coprocessor을 설치해 같이 사용해야 한다.

## 문자열

![[Pasted image 20260308013357.png]]

> [!note] 문자열 선언
> ```asm
> .DATA
> str0  DB  'Hello DOS',13,10,'$'  ; DOS 출력용 (int 21h fn 09h)
> str1  DB  "Hello bios",0         ; NULL 종료 문자열
> str2  DB  "ab"                   ; 61 62
> str3  DD  "ab"                   ; 62 61 00 00
> str4  DD  "a"                    ; 61 00 00 00
> ```
>
> > [!tip] 작은따옴표 `'` vs 큰따옴표 `"` — MASM에서 차이 없음
> > 두 방식 모두 동일하게 처리된다. C와 달리 문자/문자열을 구분하지 않는다.
>
> > [!note] 문자열 종료 방식 2가지
> > - **`'$'`** — MS-DOS `int 21h` fn 09h의 문자열 끝 마커. DOS 출력 전용.
> > - **`0` (NULL)** — BIOS 직접 호출 또는 C 스타일 문자열. `str1`처럼 마지막에 `,0` 추가.
> >
> > `DB`는 **쉼표로 구분하여** 문자열 리터럴과 바이트값을 자유롭게 섞을 수 있다.
> > ```asm
> > str0  DB  'Hello DOS',13,10,'$'
> > ;           문자열      CR LF  DOS종료
> > ```
> > `13`(CR), `10`(LF)처럼 키보드로 직접 입력할 수 없는 제어 문자를
> > 10진수 아스키값으로 끼워 넣는 방식이다.
> > 
> > DB 말고 다른 디렉티브도 문법상 쉼표로 구분이 가능하나. 1바이트 단위가 아니라 리틀 엔디언 + 패딩 문제로 의도대로 동작하지 않는다.
> >  
> 
> > [!warning] DB vs DD — 문자열 저장 방식의 차이
> > `str2 DB "ab"` → `61 62` (1바이트씩 순서대로. 엔디언 무관)
> > `str3 DD "ab"` → `62 61 00 00` (`"ab"`를 4바이트 정수 `0x00006162`로 해석 후 리틀 엔디언 저장)
> > `str4 DD "a"`  → `61 00 00 00` (`0x00000061`을 리틀 엔디언으로)
> >
> > 1바이트 단위인 `DB`는 엔디언 개념이 없고 선언 순서 그대로 메모리에 나열된다.
> > `DD`로 선언하면 문자열 전체를 **다중 바이트 정수로 취급**하여 리틀 엔디언이 적용된다.
> > 문자열은 **반드시 `DB`로 선언**해야 의도한 순서로 저장된다.

## 배열 선언

![[Pasted image 20260308142859.png]]

> [!note] 문서 문법 표기법 — BNF 계열
> 
> MASM 매뉴얼(및 언어 스펙 문서 전반)에서 문법을 설명할 때 쓰는 메타 표기법이다.
> 실제 코드에 작성하는 기호가 아니다.
>
> | 표기 | 의미 |
> |---|---|
> | `<값>` | **필수**(required) 항목 — 반드시 입력해야 함 |
> | `[[값]]` | **선택**(optional) 항목 — 생략 가능 |
> | `...` | **반복**(repetition) — 앞의 패턴을 0번 이상 반복 가능 |
>
> **예시 — DUP(Duplicate) 문법 읽는 법:**
> ```
> <개수> DUP (<초기값[[,초기값]]...>)
> ```
> - `<개수>` → 필수
> - `<초기값>` → 필수, 최소 1개
> - `[[,초기값]...]` → `,초기값`을 0회 이상 반복 가능
>
> ```asm
> arry   DB  20 DUP (2)               ; 초기값 1개
> masks  DB  10 DUP (30h,20h,01h,09h) ; 초기값 4개 반복
> ```

> [!note] DB — DUP 없이 배열 선언
> 
> 초기값을 쉼표로 나열하면 연속된 바이트 배열이 선언된다.
> 
> ```nasm
> nums DB 1,2,3,4,5   ; 5바이트 배열, 각 요소 직접 지정
> ```
> 
> DUP은 **같은 값을 반복**할 때 쓰는 축약 문법이고,  
> 서로 다른 초기값이 필요하면 이처럼 직접 나열한다.
> 
> > `DB 1,2,3,4,5` = `DB 1` `DB 2` `DB 3` `DB 4` `DB 5` 를 연속 선언한 것과 동일.

> [!note] `?` — 초기화하지 않음 (가비지 값)
> 
> ```nasm
> buffer DB 128 DUP (?)
> ```
> 
> `?`는 **"초기값을 지정하지 않겠다"** 는 의미.  
> 어셈블러가 해당 바이트를 초기화하지 않으므로,  
> 실행 시점에 메모리에 남아 있던 값이 그대로 들어간다 → **가비지 값**.
> 
> > 런타임에서 읽기 전에 반드시 직접 값을 써야 한다.  
> > C의 `char buffer[128];` (초기화 없는 지역변수) 와 동일한 개념.

```asm
.DATA
nums DB 1,2,3,4,5
mk1 DB 0A1h
array DB 20 DUP (2)
mk2 DB 0A2h
masks DB 10 DUP (30h,20h,01h,09h)
mk3 DB 0A3h
str0 DB 32 DUP ("Hello, x86!")
mk4 DB 0A4h
ddd DD 3 DUP (3 DUP (3 DUP (0)))
mk5 DB 0A5h
buffer DB 128 DUP (?)
```

마커를 추가하고 메모리 덤프를 해보았다.

![[Pasted image 20260308180216.png]]

> [!note] 직접 검증 — `DD`와 DUP의 메모리 크기 계산
> 
> **`ddd DD 3 DUP (3 DUP (3 DUP (0)))` 는 몇 바이트인가?**
> 
> | 단계 | 계산 |
> |------|------|
> | 중첩 전개 | 3 × 3 × 3 = **27개**의 DWORD |
> | DD = 4바이트 | 27 × 4 = **108바이트** |
> 
> DEBUG 덤프에서 `A4` 마커(`mk4 DB 0A4h`) 직후부터  
> `A5` 마커(`mk5 DB 0A5h`)까지 `00`이 정확히 **108개** 확인됨.
> 
> > **핵심**: `DD 0` 하나는 메모리에 `00 00 00 00` (4바이트).  
> > DUP 개수는 *요소(DWORD) 수*이고, 실제 바이트는 거기에 **×4** 해야 한다.  
> > 덤프의 `00` 개수를 셀 때 바이트 단위로 세게 되므로 헷갈리지 말 것.

`ddd DD 3 DUP (3 DUP (3 DUP (1)))` 로 코드를 변경해서 확인해보자.

![[Pasted image 20260308180733.png]]

> [!note] 직접 검증 — 리틀 엔디언과 DD의 메모리 저장 방식
> 
> **`ddd DD 3 DUP (3 DUP (3 DUP (1)))` 로 변경 후 덤프 확인**
> 
> | 선언 | 메모리 (바이트 순서) | 해석 |
> |------|----------------------|------|
> | `DD 0` | `00 00 00 00` | DWORD 0 |
> | `DD 1` | `01 00 00 00` | DWORD 1, 리틀 엔디언 |
> 
> x86은 **리틀 엔디언(Little Endian)** — 낮은 바이트가 낮은 주소에 먼저 저장됨.  
> `1`을 4바이트로 표현하면 `00 00 00 01`이지만,  
> 메모리에는 **역순**으로 `01 00 00 00` 으로 기록됨.
> 
> 덤프에서 `A4` ~ `A5` 마커 사이에  
> `01 00 00 00` 패턴이 **27개** 반복된 것으로 확인.
> 
> > **핵심**: DD 값을 덤프로 읽을 때는 4바이트를 **역순으로** 조합해야 실제 값이 나온다.  
> > `01 00 00 00` → `0x00000001` = 1

## 변수에 라벨 달기

![[Pasted image 20260308181842.png]]

> [!note] LABEL — 동일 메모리를 다른 크기로 접근하기
> 
> ```nasm
> barray LABEL BYTE   ; array와 동일 (1바이트 단위)
> warray LABEL WORD   ; array를 60개의 word로 접근 (2바이트)
> darray LABEL DWORD  ; array를 30개의 doubleword로 접근 (4바이트)
> qarray LABEL QWORD  ; array를 15개의 quadword로 접근 (8바이트)
> farray LABEL FWORD  ; array를 20개의 farword로 접근 (6바이트)
> tarray LABEL TBYTE  ; array를 12개의 10-byte로 접근 (10바이트)
> array  DB 120 DUP (?) ; 실제 메모리 선언 (120바이트)
> ```
> 
> **LABEL은 메모리를 점유하지 않는다.**  
> 새로운 변수를 만드는 게 아니라, 바로 다음에 오는 심볼의 주소를  
> 다른 크기 단위의 포인터로 재해석하는 별칭(alias)이다.
> 
> C의 포인터 캐스팅과 동일한 개념:
> 
> ```c
> char  array[120];
> short* warray = (short*)array;   // 60개
> int*   darray = (int*)  array;   // 30개
> ```


![[Pasted image 20260308194427.png]]

LABEL 뒤의 **형 지정자**가 곧 "이 주소를 몇 바이트 포인터로 볼 것인가"를 결정한다.

### 라벨에 접근하기

```asm
TITLE Main

  

.DOSSEG

.8086

.NO87

.MODEL TINY

  

.DATA

mk1 DB 0AAh

barray LABEL BYTE

warray LABEL WORD

darray LABEL DWORD

qarray LABEL QWORD

farray LABEL FWORD

tarray LABEL TBYTE

array DB 120 DUP (?) ; 120 바이트 크기의 배열 선언

mk2 DB 0BBh

  

.CODE

.STARTUP

; 디버거에서 여기서 멈추고 싶으면 아래 줄을 활성화

int 3

  

; --- BYTE 단위로 쓰기 ---

mov barray[0], 1 ; 0번째 바이트 = 01

mov barray[1], 0 ; 1번째 바이트 = 00

  

; --- WORD 단위로 읽기 ---

; 위에서 쓴 01 00 을 WORD로 읽으면 리틀엔디언으로 0x0001

mov ax, warray[0] ; ax = 0001h

  

; --- WORD 단위로 쓰기 ---

mov warray[2], 0102h ; 3번째 바이트=02, 4번째=01 (리틀엔디언)

  

; 프로그램 종료

mov ah, 4Ch

xor al, al ; 리턴값 0

int 21h

END
```

`mov ax, warray[0] ; ax = 0001h`

**A 레지스터 전체**에 워드(2바이트)단위로 값을 저장했다. 

"AX=0001"을 확인할 수 있다. 

리틀 엔디언이다. 메모리에는 "01 00"으로 저장되어 있지만, CPU에서 워드 단위로 읽어 A 레지스터에 저장할 때는 낮을 자리수인 00부터 저장된다.

warray니 2바이트(워드)단위로 읽는다.

![[Pasted image 20260308201338.png]]

`mov warray[2], 0102h ; 3번째 바이트=02, 4번째=01 (리틀엔디언)`

warray니워드 단위로 배열에 값을 썼다.

리틀 엔디언이 적용되어 `warray[2] → 시작주소 + 2바이트 → array[2]~[3]` 즉 3번째, 4번째 바이트에 02 01 이 써졌다.

> [!note] MASM `[]` — 요소 번호가 아닌 바이트 오프셋
> 
> ```nasm
> warray[0]  ; 시작주소 + 0바이트 → WORD 읽기/쓰기
> warray[2]  ; 시작주소 + 2바이트 → 다음 WORD
> warray[4]  ; 시작주소 + 4바이트 → 그 다음 WORD
> ```
> 
> | 오프셋 | 접근 바이트 | C# 등가 |
> |--------|------------|---------|
> | `warray[0]` | array[0] ~ array[1] | `warray[0]` |
> | `warray[2]` | array[2] ~ array[3] | `warray[1]` |
> | `warray[4]` | array[4] ~ array[5] | `warray[2]` |
> 
> **LABEL의 형(WORD, DWORD 등)은 한 번에 읽고 쓰는 크기만 결정한다.**  
> `[]` 안의 숫자는 항상 **바이트 단위** 이동이다.
> 
> > C#의 `array[i]` → i번째 **요소**  
> > MASM의 `label[n]` → 시작 주소에서 n**바이트** 떨어진 위치  
> > WORD 단위로 순회하려면 `0, 2, 4, 6 ...` 으로 **타입 크기만큼 증가**시켜야 한다.

## 포인터 변수

![[Pasted image 20260308203547.png]]

> [!note] 포인터 변수 — near / far
> 
> ```nasm
> msg     DB "Hello, x86!", 0
> nearptr DW msg   ; 2바이트 — offset만 저장
> farptr  DD msg   ; 4바이트 — offset + segment 저장
> ```
> 
> | 종류 | 선언 | 크기 | 저장 내용 |
> |------|------|------|-----------|
> | near 포인터 | `DW` | 2바이트 | offset만 |
> | far 포인터  | `DD` | 4바이트 | offset(2) + segment(2) |
> 
> x86-16에서 심볼의 주소 원본은 항상 **segment:offset (32비트)** 쌍이다.  
> `DD msg`는 이 주소를 있는 그대로 저장하는 **기본 동작**이고,  
> `DW msg`는 MASM이 offset만 꺼내서 저장하는 **특별한 축약**이다.
> 
> > **왜 DW가 offset만 저장하는가?**  
> > near 포인터는 "같은 세그먼트 안에서만 사용한다"는 전제가 깔려 있다.  
> > segment는 이미 DS/CS 레지스터에 있으므로, offset만으로 주소를 완성할 수 있다.
> 
> **far 포인터 메모리 레이아웃 (리틀 엔디언)**
> ```
> [0] [1]   [2] [3]
>  offset    segment
> ```
> 
> x86-16 물리 주소 계산:
> ```
> 물리 주소 = segment × 16 + offset
> ```
> 16비트 + 16비트 조합으로 **20비트 물리 주소**를 만든다.

참고로 TINY 모델에서는 PPP의 코드는 실행이 불가능하다.

> [!QUOTE] `A2118: cannot have segment address references with TINY mode`
> 
> `.MODEL TINY` 는 **코드+데이터+스택이 전부 같은 세그먼트** 안에 들어가는 모델입니다 (`.COM` 파일).
>  
> 세그먼트가 하나뿐이라서 **segment 값이 의미가 없습니다.**
