# MS-DOS 부팅 순서 4~6

## MS-DOS의 부팅 순서 4

![[Pasted image 20260305015912.png]]

> [!note] 부팅 가능한 장치가 있는 경우
> 
> [[005. MS-DOS 부팅 순서 1~3#MS-DOS의 부팅 순서 3]]에서 언급한대로 부팅 가능한 장치(예: MS-DOS 2.0 플로피 디스크)가 있다고 가정한다.

> [!info] ROM 로더란?
> **ROM BIOS 안에 내장된 작은 프로그램**으로, 역할은 딱 하나다.
> 
> 부팅 가능한 장치(디스크 등)를 찾으면, 그 장치에 저장된 **운영체제 로더(boot sector)를 RAM에 복사**하고 해당 위치로 **점프**한다. 
> 
> 제어권이 운영체제 로더로 넘어간 뒤 ROM BIOS가 진행하는 초기화 루틴은 끝이다.
> 
> > [!tip] 왜 복사하는가?
> > CPU는 디스크의 코드를 직접 실행할 수 없다.
> > **RAM/ROM에 올라온 코드만** 실행 가능하기 때문에 복사가 필요하다.

> [!info] boot sector는 왜 low memory가 아닌 위쪽에 복사되나?
> 메모리 다이어그램을 보면 boot sector는 **low memory(하단)가 아닌 위쪽 RAM 영역**에 복사된다.
>
> > [!tip] 이유 — 어차피 사라질 코드이기 때문
> > boot sector(운영체제 로더)는 **OS를 메모리에 올리는 임시 역할**만 하고 끝난다.
> > 실행이 끝나면 더 이상 필요 없으므로, 굳이 소중한 **low memory를 차지할 필요가 없다**.
>
> > [!warning] low memory는 아껴야 한다
> > `0x00000` ~ `0x00400` 인터럽트 벡터, ROM BIOS 데이터 등
> > **지속적으로 필요한 데이터**가 low memory에 자리잡고 있기 때문이다.

> [!info] ROM BIOS 데이터는 OS 이후에도 살아있다
> `0x00400` 영역의 ROM BIOS 데이터는 부팅 중에만 쓰이는 게 아니라,
> **OS가 뜬 이후에도 BIOS 인터럽트 호출 시 계속 참조**된다.
>
> 따라서 boot sector처럼 "쓰고 버리는" 코드와 달리,
> **low memory에 영구적으로 자리를 차지**하는 것이 당연하다.

## MS-DOS의 부팅 순서 5

![[Pasted image 20260305020407.png]]

> [!info] 용어 정리 — PPT 표현 vs 실제 의미
> | PPT 표현 | 실제 의미 |
> |---|---|
> | 운영체제 로더 | boot sector |
> | 다음 부분 | `IO.SYS`, `MSDOS.SYS` |

> [!note] boot sector의 역할
> boot sector(운영체제 로더)는 플로피 디스크에 있는 **`IO.SYS`**, **`MSDOS.SYS`** 를 RAM에 복사한다.
>
> > [!warning] boot sector는 직접 디스크를 읽을 권한이 없다
> > 디스크에서 파일을 읽어오는 기능은 **ROM BIOS에 구현**되어 있기 때문에,
> > boot sector는 직접 처리하지 못하고 **ROM BIOS의 기능을 빌려서** 사용한다.
>
> > [!tip] ROM BIOS 기능에 접근하는 방법 — 인터럽트 벡터
> > ROM BIOS의 기능을 호출하려면 **인터럽트 벡터 테이블**에서 필요한 함수 주소를 찾아 호출한다.
> > 이것이 뒤에서 배울 **소프트웨어 인터럽트** 개념이다.

> [!info] ROM BIOS는 왜 계속 메모리에 있어야 하는가? 
 boot sector가 실행되는 동안 ROM BIOS의 기능을 계속 호출해야 하기 때문에, 
  **ROM BIOS는 부팅 내내 메모리에 떠있어야 한다.**

> [!success] boot sector의 마지막 — 제어권 넘기고 퇴장
> `IO.SYS` / `MSDOS.SYS`를 모두 RAM에 올린 뒤,
> boot sector는 해당 코드로 **제어권을 넘기고 메모리에서 제거**된다.
> 여기까지가 boot sector(운영체제 로더)의 역할 전부다.


## MS-DOS의 부팅 순서 6

![[Pasted image 20260305021144.png]]

> [!info] SYSINIT이란?
> `IO.SYS` / `MSDOS.SYS` 안에는 **부팅 중에만 필요한 코드**와 **계속 필요한 코드**가 섞여 있다.
> 이 중 **부팅이 끝나면 필요 없는 코드**를 **SYSINIT**이라고 한다.
>
> - SYSINIT은 위쪽(high memory) 으로 복사
> - `IO.SYS`의 나머지(계속 필요한 코드)는 압축하여 low memory에 재배치
> - 이것이 PPT에서 말하는 **"메모리 위치를 바꿈"**

> [!note] SYSINIT이 하는 일 — 인터럽트 벡터 업데이트
> 원래 인터럽트 벡터는 **ROM BIOS의 함수**를 가리키고 있었다.
> SYSINIT 실행 중에 **일부 항목을 `MSDOS.SYS`(DOS 커널)의 함수로 교체**한다.
>
> > [!tip] 왜 교체하는가?
> > ROM BIOS보다 **DOS가 직접 제공하는 기능**이 더 풍부하고 OS 수준에 맞기 때문.
> > 이후 어떤 프로그램이 해당 기능을 호출하면
> > 인터럽트 벡터 → `MSDOS.SYS` 코드로 점프하여 실행된다.

> [!info] IO.SYS — 장치 드라이버의 집합
> `IO.SYS`에 남아있는 코드는 **하드웨어를 직접 제어하는 장치 드라이버**다.
>
> > [!warning] 왜 MS-DOS가 직접 못 만드나?
> > 하드웨어마다 제어하는 어셈블리어 코드가 다르기 때문에,
> > **MS-DOS(운영체제) 측에서는 하드웨어 구조를 알 수 없다.**
> > 따라서 **하드웨어 제조사가 장치 드라이버를 직접 제공**해야 하고,
> > 이 코드가 `IO.SYS`에 담긴다.

"메모리 위치를 바꿈", "인터럽트 벡터 일부를 업데이트" 이 과정을 모두 포함하는 것이 "DOS 초기화"이다.

![[Pasted image 20260305023008.png]]

 > [!success] 마지막 단계 — 셸(Shell) 실행
> SYSINIT은 마지막으로 **`COMMAND.COM`(셸)을 RAM에 복사**하고 제어권을 넘긴다.
> 이 시점부터 MS-DOS 부팅이 완료된 것이다.
>
> > [!info] 메모리에 쌓이는 순서
> > ```
> > IO.SYS
> > MSDOS.SYS
> > MS-DOS 관련 데이터
> > COMMAND.COM      ← 여기로 최종 제어권 이동
> > ```
>
> > [!tip] COMMAND.COM의 역할
> > 이제부터 모든 권한은 **`COMMAND.COM`이 가진다.**
> > 사용자가 명령어를 입력하면 `COMMAND.COM`이 받아서 처리하는
> > **유저 인터페이스(셸)** 역할을 한다.
