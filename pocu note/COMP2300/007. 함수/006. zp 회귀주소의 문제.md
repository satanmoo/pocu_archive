# zp 회귀주소의 문제

![[Pasted image 20260226202148.png]]

`retaddr`를 하나만 두고 복귀 주소를 관리하면, 호출이 한 단계일 때는 동작하지만 재귀처럼 중첩 호출될 때는 실패한다. 

원인과 해결 방향을 정리한다.

## 핵심 정리

- `retaddr`는 실행 시점에 **한 개의 주소**만 저장한다.
- 중첩 호출이 생기면 바깥 호출의 `retaddr`가 안쪽 호출에 의해 덮어 써진다.
- 따라서 최상위 호출이 아닌 중간/안쪽 복귀 지점으로 점프하게 된다.
- `jmp (retaddr)`는 인디렉션 방식 자체는 해결했지만, **복귀 저장소 개수(스택 미제공)** 문제가 남는다.

## 문제 2: 재귀 탈출 조건이 있어도 안 풀리는 이유

![[Pasted image 20260226203025.png]]

![[Pasted image 20260226203042.png]]

돌아올 주소(`end`)를 `retaddr`에 저장한 뒤 `func`을 실행한다.

![[Pasted image 20260226203139.png]]

재귀 탈출 조건을 만족 못 했다고 가정하고 진행하면,  `retaddr`에 `ret`(재귀 호출이 끝나고 돌아와서 실행할 주소)가 저장된다.

`ret`으로 `jmp (retaddr)`를 수행하면 나중에 재귀 탈출 조건을 만족하더라도 `ret`으로 되돌아가는 무한 루프가 생긴다.  

따라서 호출자가 본래의 `end`로 복귀하지 못한다.

> [!NOTE] `jmp (retaddr)`만으로 해결되지 않는 이유
> 
> `jmp (retaddr)`는 절대 주소에 고정된 점프를 간접 참조로 바꾼 것이라 `retaddr`의 값이 가리키는 주소로 점프한다.
> 
> 하지만 중첩 호출에서는 **복귀 주소 저장소가 하나뿐**이므로  최신 호출의 값으로 계속 덮어 씌워져 스택처럼 동작하지 않는다.


## 땜빵 1: 호출 깊이 별 `retaddr` 분할

![[Pasted image 20260226203605.png]]

함수 깊이에 따라 `retaddr`을 따로 두는 방법.

![[Pasted image 20260226203807.png]]

각 깊이에 `retaddr0`, `retaddr1` … 를 미리 정해 두고 사용하면 작동한다.  

`retaddr`를 전역에서 미리 합의한 슬롯으로 분리한 형태이다.

### 한계

- 댐(depth)이 깊어질수록 제로 페이지 변수 선언이 늘어난다.
- 재귀 깊이를 컴파일 타임에 제한해야 하며, 실전적인 확장성이 없다.
- 즉, 임시 대응일 뿐 정식 해법이 아니다.

## 정식 해법 방향: 스택 기반 복귀 주소

6502에서 정석적으로 함수 호출/복귀를 구현할 때는 보통 `JSR`/`RTS`를 사용한다.
`JSR`은 호출 지점의 복귀 주소를 하드웨어 스택에 저장하고, `RTS`가 LIFO 순서로 꺼내어 복귀한다.

- 정적 `retaddr` 하나로 생기는 덮어쓰기 문제를 피할 수 있다.
- 재귀 함수도 스택 깊이만큼 자연스럽게 처리된다.
- 스택은 하드웨어 동작을 이용하므로 호출-복귀 규칙이 일관된다.

> [!SUMMARY]
> `jmp (retaddr)`은 회귀 주소를 간접으로 해석하는 문제는 풀어주지만,  재귀/중첩 호출 자체를 설계 레벨에서 지원하지 못한다. 
> 
> 재귀가 필요한 함수는 **반드시 스택 기반 호출 규약**으로 가야 한다.
