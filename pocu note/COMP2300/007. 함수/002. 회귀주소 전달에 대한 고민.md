## 점프할 주소에 대한 고민 - 사례 1

### caller

![[Pasted image 20260224111245.png]]

> [!INFO] caller 코드 분석
> - `ldy #0` : Y 레지스터를 0으로 초기화
> - `jmp print` : `print:` 레이블로 점프

호출자에서 점프(`jmp`)는 명확하다. 목적지가 `print:` 레이블 하나이기 때문이다.

### callee

![[Pasted image 20260224111520.png]]

> [!INFO] callee(print:) 코드 분석
> - `lda #'.'` : `.` 문자를 A 레지스터에 로드
> - `sta out` : A를 출력 포트에 저장
> - `iny` : Y 레지스터 증가
> - `sty column` : Y를 `column` 변수에 저장
> - `jmp retaddr` : 회귀주소로 점프

> [!QUESTION] 회귀 주소를 어디에 저장할까?
> caller가 누구인지에 따라 회귀 주소는 달라진다.
> callee는 어디로 돌아가는지 회귀 주소를 알고 있어야 한다.
> 결과적으로 **caller가 callee에게 회귀 주소를 알려주게 된다**.

## zp에 회귀주소를 저장

![[Pasted image 20260224111922.png]]

> [!NOTE] 아이디어
> caller가 `retaddr`에 돌아갈 주소를 저장하고 callee를 호출한다.
> `retaddr`는 zero page에 위치한 2바이트 변수다.

![[Pasted image 20260224112009.png]]

> [!INFO] 회귀주소 저장 코드 (리틀 엔디언)
> ```
> ldx #<op2      ; op2 주소의 하위 바이트
> stx retaddr    ; retaddr에 저장
> ldx #>op2      ; op2 주소의 상위 바이트
> stx retaddr+1  ; retaddr+1에 저장
> ```
> - `#<op2` : `op2` 레이블 주소의 하위 바이트(low byte)
> - `#>op2` : `op2` 레이블 주소의 상위 바이트(high byte)
> - 리틀 엔디언 방식으로 `retaddr`(low), `retaddr+1`(high) 순서로 저장

회귀주소가 `op2`이다. `print:` 레이블을 실행하고 `jmp retaddr`로 회귀주소로 돌아간다.

![[Pasted image 20260224112728.png]]

> [!WARNING] 실행 결과가 올바르지 않다!
> 이 방식으로 구현해도 정상적으로 동작하지 않는다. 왜 안 될까?

스파이더를 끼우고 확인해보자.

![[Pasted image 20260224113219.png]]

> [!INFO] 회귀 주소 저장 확인 (스파이더 버스 모니터)
> `retaddr = $80`이므로:
> - `$0080` : `1D` (op2 주소의 low byte)
> - `$0081` : `80` (op2 주소의 high byte)
>
> 리틀 엔디언으로 조합하면 → `$801D` (= `op2` 레이블의 주소)

![[Pasted image 20260224113459.png]]

> [!INFO] `jmp print` opcode 분석
> [[009. jmp의 주소지정 모드#상세 동작표 jmp (절대 주소지정 모드)]] 참고
> - `4C` : `jmp`의 opcode (절대 주소지정 모드)
> - `5E` : `print:` 주소의 low byte
> - `80` : `print:` 레이블 주소의 high byte
>
> → `$805E`로 분기

![[Pasted image 20260224172714.png]]

> [!INFO] print: 실행 흐름
> `$805E`로 분기해서 `lda #'.'`을 실행한다.
> - `A9` : `lda`의 opcode

![[Pasted image 20260224172809.png]]

`lda`부터 쭉 이어서 실행한다.

![[Pasted image 20260224172901.png]]

> [!INFO] `jmp retaddr` opcode 분석
> - `4C` : `jmp`의 opcode (절대 주소지정 모드)
> - `80` : `retaddr`의 low byte
> - `00` : `retaddr`의 high byte
>
> → **`$0080`으로 점프**한다.

![[Pasted image 20260224172959.png]]

> [!WARNING] 문제 발생!
> `$0080`으로 점프해서 `1D`를 **opcode로** 읽는다.
>
> 우리의 의도는 `$0080`에 저장된 값(`1D 80`)을 **주소로 해석해** `$801D`로 돌아오는 것이었는데,
> `$0080` 자체로 점프해서 `1D`를 명령어로 해석하게 된다.

### 왜 잘못됐을까?

> [!BUG] 절대 주소지정 모드의 함정
> `jmp retaddr`는 **절대 주소지정 모드**(`4C`)로 어셈블된다.
>
> | 의도 | 실제 동작 |
> |------|----------|
> | `$0080`에 저장된 값(`1D 80` → `$801D`)으로 점프 | `$0080` 주소 자체로 점프 |
>
> - **절대 주소 모드**: 어셈블 시점에 주소가 `$0080`으로 고정된다 (어셈블러의 역할)
> - **우리의 필요**: 실행 시점에 `$0080`에 저장된 값을 주소로 사용해야 한다

![[Pasted image 20260224173845.png]]

> [!DANGER] 절대 주소지정 모드 사용 불가!
> `jmp retaddr`를 사용하면 `retaddr` 변수의 **주소**(`$0080`)로 점프한다.
> 우리가 원하는 것은 `retaddr`에 **저장된 값**(`$801D`)으로 점프하는 것이다.
> 따라서 절대 주소지정 모드는 이 용도에 사용할 수 없다.

![[Pasted image 20260224173925.png]]

> [!SUCCESS] 해결책: 간접 참조!
> **간접 주소지정 모드(Indirect Addressing Mode)**를 사용하면 된다.
> `jmp (retaddr)` 형태로 작성하면, `retaddr`에 저장된 값을 주소로 사용해 점프한다.
