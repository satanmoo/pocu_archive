## 페이지 크로싱 (Page Crossing)

![[Pasted image 20260131210253.png]]

> adc nums,X

nums 가 베이스 주소이다. 이게 `0xC0FC`

"X 레지스터"의 값은 2다. 따라서 베이스 주소에 2를 더하면 `0xC0FE`가 된다.
- `0b00000010`

> 페이지가 바뀌지 않는 경우

`0xC0FE` 는 페이지를 넘어가지 않았다.
- 페이지 범위는 "0xC000" ~ "0xC100" 이다.

![[Pasted image 20260131210658.png]]
> X 레지스터 00000110

"X 레지스터"의 값이 6으로 바뀌었다.

따라서 "색인 주소지정 모드"로 계산한 주소는 `0xC102`로 페이지가 바뀌었다.
- 페이지를 바꾸는 동작은 CPU 내부에서 해준다.
- 어셈블리어로 코드로 제어해주는 것이 아니라 "페이지 크로싱"이 발생했다고 보면 된다.
- 즉 실행 중에 조건("페이지 크로싱")에 따라 CPU 사이클 수가 늘어났다.
	- 하드웨어의 동작

![[Pasted image 20260131210951.png]]

> 하위 주소의 덧셈(BAL + X) 에서 받아올림이 생김

위의 예시에서 보면 `0xC0FE` 에서 하위 주소(BAL)는 `0xFE`다.  "X 레지스터"의 값 6을 더하면 받아올림이 생긴다.

> 그 받아올림을 상위 주소에 더해야함(BAH + C)

상위 주소(BAH)는 `0xC0`이다. 받아올림 결과가 더해저 상위 주소는 `0xC1`이 된다.

### 페이지가 바뀌지 않는 경우

![[Pasted image 20260131211634.png]]
받아올림이 발생하지 않으면 바로 `Fetch data1`을 수행할 수 있다.
- 내부 연산도 비어있다.
	- 페이지 크로싱이 발생하면 빈 내부 연산에 BAH + C 계산이 들어가고, `Fetch data1`은 다음 사이클에 연산한다.

![[Pasted image 20260131211835.png]]
> 최종 주소를 찾으려면 1 사이클이 추가로 필요

본질적으로 BAH + C 때문에 발생한다.

## zp,x가 여전히 4 사이클인 이유

![[Pasted image 20260131212136.png]]

> a,x 모드에서 BAH fetch는 공짜

표를 보면 BAL + X 하면서 BAH fetch를 동시에 수행한다.

하지만 "zp,x 주소 지정 모드"는 BAH를 읽어오지 않는다. 따라서 그냥 한 사이클 손해보는 개념이다.

![[Pasted image 20260131212446.png]]

> 3 사이클 "Fetch byte"

"BAL + X"를 계산하면서 Fetch 하지만 쓸모 없다.

### zp,x 가 여전히 4 사이클인 이유

![[Pasted image 20260131212618.png]]

> BAL + X 계산은 피할 수 없음

이 계산해야지 주소가 확정이 된다.

페이지 크로싱을 발생하지 않는다. "zp 주소 지정모드"에서는 wrap around 되는 개념으로 받아 올림이 없다!
- "제로 페이지"니까 원 페이지로 넘어가지 않는다고 생각하면 된다!

> [!NOTE] "여전히" 4사이클인 이유는 페이지 크로싱이 일어나지 않기 때문이다. "4사이클"인 이유는 BAL + X를 피할 수 없기 때문이다.
