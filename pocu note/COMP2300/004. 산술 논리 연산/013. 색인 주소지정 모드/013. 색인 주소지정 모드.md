## 색인 주소지정 모드

![[Pasted image 20260131174839.png]]

> 피연산자 주소에 더해 줌

"주소 지정 모드" 표기 중에 ","(컴마)가 달린 것들이다.

각 주소에서 색인 레지스터에 있는 값을 더한 주소를 지정한다.
- 포인터 개념

### 절대 주소지정 모드 코드

![[Pasted image 20260131175041.png]]

> [!TIP] [[022. 데이터 블록]]을 참고하자.

`%C000` :  `$00`
`%C001` :  `$01`
`%C002` :  `$02`
`%C003` :  `$03`

위와 같이 메모리에 저장된다.

> [!NOTE] "adc 절대 주소 모드"를 사용한다.

"adc 절대 모드" 나 "adc 절대 색인 모드"나 사이클이 동일하다. 따라서 `ldx`, `inx`에서 사이클 손해가 발생한다.

### 그러면 색인 주소지정 모드를 왜 쓰는가?

코드 작성 시점에 반복문과 같이 동적으로 얼마나 주소를 지정할 지 모르는 경우에 효용이 있다.
- 색인 주소지정 모드 밖에 사용할 수 없다.

위 예시처럼 정적으로 4개 지정은 "색인 주소지정 모드"를 사용하지 않는 것이 좋다.

## 상세 동작표: a,x 주소지정 모드

![[Pasted image 20260131181612.png]]

> BAL, BAH

base adress
- 위에서 `nums`같은 것을 의미한다.

> 외부연산: Fetch opcode1

아직 이 연산을 모른다.
- 당연히 "a,x 주소지정"모드 인지 모른다.

> 외부연산: Fetch BAL, 내부연산: Decode opcode1

파이프라인(overlapping) 개념으로 동시에 명령어 처리

opcode1을 해석해 "a,x 주소지정 모드"임을 인지한다.

"a,x 주소지정 모드"가 확정이라 BAL이 유효한 값이다.

> 외부연산: Fetch BAH, 내부연산: BAL + X 계산

BAL만 읽어도 X 레지스터의 색인 값과 더하는데 문제가 없구나?!
- 둘다 8비트니까!

BAH를 읽는 동안 동시에 미리 BAL과 X 레지스터의 색인 값을 더해놓는다.

> Fetch data1

BAH 과 "BAL + X 레지스터" 조합의 주소에 있는 데이터를 읽어온다.
- "a,x 주소지정 모드"
