## 배열

![img_41.png](images/img_41.png)

- 가정:
    - 배열의 원소의 순서를 바꾸지 않아야 한다는 가정

- 삽입
    - 제일 마지막 빈 위치에 삽입하는 경우:
        - O(1)
        - 최선
    - 제일 앞에 삽입하는 경우:
        - O(N)
        - 최악
        - 밀어야함
- 삭제
    - 제일 마지막 요소 삭제:
        - O(1)
        - 최선
    - 제일 앞에 요소 삭제:
        - O(N)
        - 최악
        - 삭제하고 앞의 인덱스로 당겨야함
- 검색
    - 매 요소 순회하면서 특정 값인지 비교
    - O(N)

## 스택

![img_42.png](images/img_42.png)

- 삽입:
    - Top 위치에 삽입
    - O(1)
- 삭제:
    - Top 위치에 제거
    - O(1)
    - 스택의 삭제와 삽입은 배열의 맨 끝에서 삭제와 삽입과 다를게 없음
- 검색:
    - 원칙상 찾을 때 까지 모두 뽑아서 다른 곳에 저장했다가, 다시 복원
    - O(2N)

## 큐

![img_43.png](images/img_43.png)

- 삽입:
    - Back 위치에 삽입
    - O(1)
- 삭제:
    - Front 위치에서 제거
    - O(1)
- 검색:
    - 원칙상 찾을 때 까지 모두 뽑아서 다른 곳에 저장했다가, 다시 복원
    - O(2N)

## 연결 리스트

![img_44.png](images/img_44.png)

- 검색:
  - 모두 순회
  - O(N)
- 삽입, 삭제:
  - 추가, 지우고 다시 연결하기
  - O(1)

- 삽입과 삭제에서 이론적으로 유리해도 실제로 대부분 배열이 빠름
  - 노드를 메모리에 생성하고 삭제하는 과정이 오버헤드가 크기 때문
    - 이를 보완하기 위해 메모리 풀(memory pool) 방식으로 미리 메모리를 할당해서 재활용하는 방식으로 최적화

## 해시 테이블, 해시 맵

![img_45.png](images/img_45.png)

- 해시 함수에서 계산하는 시간은 입력(N)에 영향을 받지 않음
- 버킷(배열)에 인덱스가 충돌하는 경우:
  - 실제로 해시 충돌
  - 해시값을 다른데 인덱스가 충돌
    - 링크드 리스트로 구현

- 최적의 버킷사이즈는 보통 2N + a
  - 공간복잡도 O(N)

![img_46.png](images/img_46.png)

- key, value 배열을 각각 총 2개 만들고, key로 해시함수 + 인덱스 구해서 인덱스에 저장
