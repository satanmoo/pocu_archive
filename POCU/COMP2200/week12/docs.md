![img.png](img.png)

- 소스 코드 공개 없이 라이브러리 배포 가능
    - 사용자가 라이브러리의 함수를 사용하기 위해서 함수 시그니처가 포함된 `헤더 파일`은 공개해야함
    - 오브젝트 파일의 모음은 소스 코드가 아님, 이진 파일(.o확장자, 오브젝트 파일)

## 정적 라이브러리

### [정적 라이브러리와 링크]

![img_1.png](img_1.png)

- 정적 링킹
    - 컴파일 하는 도중에 결정
    - 라이브러리 안의 기계어를 최종 실행 파일에 복사
- 동적 링킹에 비해
    - 실행 파일의 크기가 커짐
    - 메모리를 더 차지함
    - 실행 속도는 빠름
        - 합칠 때 링커에서 최적화가 들어감
- 링커는 그림에서 `오브젝트 파일`에 `정적 라이브러리의 기계어`를 복사함

### [정적 라이브러리 사용하기]

![img_2.png](img_2.png)

- 아래 3단계를 따름
- [1] 소스 코드들을 컴파일하여 정적 라이브러리를 만듦
    - 파일 하나(오브젝트 파일)
    - 여러 프로젝트면 오브젝트 파일 여러 개
    - 확장자
        - *.lib: 윈도우 비쥬얼 스튜디오
        - *.a: 리눅스 계열(아카이브 라고 부름)
- [2] 다른 소스 코드들을 작성하고, 컴파일할 때 라이브러리의 헤더 파일들을 사용
    - 작성하는 소스 코드는 실행 파일을 만들기 위해 보통 main()함수를 가지고 있음
    - 정적 라이브러리와 작성하는 소스 코드는 독립적임
        - 정적 라이브러리의 코드를 외부에서 가져다 사용하는 개념
- [3] 작성한 소스코드와 정적라이브러리 링킹

## 동적 라이브러리

![img_15.png](img_15.png)

- 동적 링킹
    - 동적 라이브러리와 링크하는 것
- 실행파일 안에 여전히 구멍을 남겨둠
- 실행파일을 실행할 때 실제로 링킹이 일어남
    - 링킹의 주체는 운영체제

![img_16.png](img_16.png)

- 정적 링킹에 비해
    - 실행 파일 크기가 작음
    - 메모리 절약
- DLL 지옥

### [동적 라이브러리 사용하기]

![img_17.png](img_17.png)

- [1] 소스 코드들을 컴파일하여 동적 라이브러리를 만듦
    - 윈도우에서 확장자 *.dll
    - 리눅스 계열에서 확장자 *.so
- [2] 다른 소스 코드들을 작성할 때 동적 라이브러리의 헤더 파일들을 사용
- [3] 컴파일 할 때 동적 라이브러리와 함께 `링킹`
    - 동적 라이브러리에 있는 기계어가 실행파일에 포함되지 않음
    - 실행 중에 동적으로 링킹할 수 있는 정보만 포함
    - 실행 파일이 정상적으로 작동하기 위해서는 동적 라이브러리 파일과 함께 배포되야함
        - 유저 컴퓨터에 동적 라이브러리 파일이 있어야함
        - 정적 라이브러리와 달리 실행 파일에 라이브러리의 코드가 포함되지 않음

## 정적 vs 동적 라이브러리

### [동적 라이브러리의 장단점]

![img_21.png](img_21.png)

- 장점
    - 실행 파일을 바꾸지 않고, 동적 라이브러리만 업데이트 가능
    - 반대로 동적 라이브러리는 바꾸지 않고, 실행 파일만 업데이트 가능
    - 필요에 따라 동적 라이브러리를 선택적으로 로딩
        - CPU 세대 별로 동적 라이브러리 파일 만들어 두고, 실행할 때 CPU 버전에 맞게 동적 라이브러리 파일 링킹
- 단점
    - 해킹 당하기 쉬움
        - DLL 인젝션
        - 함수 시그니처만 알면, 내가 만든 해킹용 DLL 파일을 로드(링킹)할 수 있음
            - 불법 소프트웨어 DLL 파일 덮어쓰기
    - DLL 지옥

### [정적 라이브러리의 장단점]

![img_22.png](img_22.png)

- 장점
    - 해킹에 안전함
        - 정확한 버전의 라이브러리가 실행파일 안에 내포
    - 최적화에 유리
        - 링커에서 최적화
            - 쉽게 생각하면 정적 라이브러리를 사용하는 소스 코드의 함수에서 정적 라이브러리의 어떤 함수를 호출하는지 알 수 있음
- 단점
    - CPU 세대 별로 실행파일을 만들어서 배포
        - 보통 가장 후지거나, 메이저한 CPU 세대에 맞춤
    - 라이브러리의 소스코드가 바뀌면 실행파일도 재배포
    - 실행파일의 크기 커짐
    - 실행 중 다른 실행파일들과 라이브러리 공유 불가

## 인라인 함수

![img_26.png](img_26.png)

- 매크로 함수의 장점과 단점을 생각해보자
- 장점
    - 함수 호출이 아니라 코드 그 자체를 복붙하니까, 함수 호출에 따른 과부하를 막을 수 있음
- 단점
    - 디버깅이 매우 힘듦
    - 가동성도 매우 떨어짐
- 매크로 함수의 장점만 뽑아보자!

![img_27.png](img_27.png)

- 함수 앞에 `ìnline` 키워드를 붙이면 됨
- 컴파일러에게 최적화 해달라고 `힌트`를 주고, 요청
    - 컴파일러가 무시할 수도 있음
    - inline 키워드가 없어도 컴파일러가 알아서 최적화 해줌

![img_28.png](img_28.png)

- 복붙을 할 수 있으려면? 조건이 있음
- 인라인 함수를 호출하는 코드를 `컴파일`할 때 그 함수의 구현을 알아야함
    - 인라인 함수가 아닌 컴파일 과정(전처리-어셈블-컴파일-링킹)과 비교해보자
    - 컴파일 과정에서 함수의 구현을 알 필요가 없었고, 함수의 선언만 필요했음
        - 링킹 과정에서 함수의 선언을 통해 구멍을 매꾸기 때문
    - 반면 인라인 함수는 함수의 구현이 `컴파일` 단계에서 필요함
        - 따라서 함수의 구현이 `헤더파일`에 들어가야함!

## 코드 보기: 정적 인라인 함수

```c++
#ifndef STATIC_MATH_H
#define STATIC_MATH_H

static inline int static_add(int a, int b)
{
    return a + b;
}

#endif /* STATIC_MATH_H */
```

- static 키워드가 붙어있어서, 파일 스코프

```c++
// adder.c
#include <stdio.h>

#include "adder.h"
#include "inline_math.h"
#include "static_math.h"

void run_adder(void)
{
    printf("1 + 2 = %d\n", static_add(1, 2));
    printf("addr of static_add: %p\n",
        (void*)static_add);

    printf("1 + 2 = %d\n", inline_add(1, 2));
    printf("addr of inline_add: %p\n", 
        (void*)inline_add);
}
```

- #include "static_math.h"를 통해서 위의 static inline 함수를 복붙
    - static inline 함수는 이 adder.c 파일 안에서만 사용할 수 있음
    - inline 키워드가 붙었기 때문에 함수 심볼을 따로 만들지 않음
      - 이 translation unit을 만들 때 쓰고, 링킹 단계에서는 사용하지 않기 때문
    - 다른 .c 파일에서 이 헤더 파일을 인클루드 해도 중복되는 심볼이 없음
        - 심볼이 생성되지 않기 때문임
    - 전처리기에서 복붙하기 때문에 다른 .c 파일에서도 static inline 함수의 구현이 생김
      - 인라인화가 되면 문제없음
      - 인라인화 실패하면 함수 호출이 필요함, 이 때 각기 다른 .c 파일에서 파일 내부의 static inline 함수를 호출
        - 이를 확인하는 것이 run_adder() 함수의 내용
- 즉 static의 효과
  - 파일 스코프에서 접근

```c++
// main.c
#include <stdio.h>

#include "adder.h"
#include "inline_math.h"
#include "static_math.h"

int main(void)
{
    printf("1 + 2 = %d\n", static_add(1, 2));
    printf("addr of static_add: %p\n", 
        (void*)static_add);

    printf("1 + 2 = %d\n", inline_add(1, 2));
    printf("addr of inline_add: %p\n", 
        (void*)inline_add);

    run_adder();

    return 0;
}
```

- 출력 결과
    - inline_add 주소 같음
        - 인라인화 안 되어 함수 호출한다는 가정
    - static_add 주소 다름
        - 각 파일마다 실제 호출하는 함수가 있음
        - 인라인화 안 되어 함수 호출한다는 가정
        - 파일의 크기만 2배로... 늘어남
- 결론! static inline 사용하지 말자!!

### [코드 해석]

- #include "adder.h"
    - void run_adder(void);의 선언을 복붙하기 위함
    - main() 내부에서 run_adder()을 호출해야하기 때문
- #include "inline_math.h"
  - 아래 인라인 함수를 복붙하기 위함
  - 인라인 함수는 심볼이 생성되지 않음
    - 인라인화 실패해서, 함수 호출이 필요할 경우 링커에서 찾을 수 없음
    - 인라인화 성공해서, 복붙하면 문제없음
    
```c++
// inline_math.h
#ifndef INLINE_MATH_H
#define INLINE_MATH_H

inline int inline_add(int a, int b)
{
    return a + b;
}

#endif /* INLINE_MATH_H */
```
  - 따라서 인라인 함수 베스트 프렉티스에 따라 아래와 같이 inline_math.c 파일이 필요함

```c++
// inline_math.c
#include "inline_math.h"

extern inline int inline_add(int a, int b);
``` 
  - #include "inline_math.h"로 인라인 함수의 선언/구현 복붙
  - extern 키워드를 인라인 함수의 선언에 붙여서 다시 선언
  - inline_math.c 파일 안에서만 함수 심볼이 생성됨
  - 인라인화 성공하면 문제없음
  - 인라인화 실패하면 함수 호출이 필요할 때 함수 심볼 하나를 찾을 수 있음
    - 따라서 링커에서 심볼 중복되서 오류 발생 X

- #include "static_math.h"
  - static inline int static_add(int a, int b); 의 선언/구현을 복붙하기 위함
  - 이 때 static 키워드가 붙었기 때문에 인라인화가 실패하면 main.c 내부의 복붙한 함수를 호출함
  - 따라서 아래 run_adder()로 adder.c 파일의 static_add 함수를 호출할 때와 주소가 다름!!!

- 나머지 코드
  - main.c 내부의 static_add() 주소와 adder.c 파일의 static_add()의 주소가 다른 것을 확인하는 코드
  - inline_add()의 주소는 inline_math.c 안의 함수의 주소

  - 컴파일러가 이 키워드에 따라 최적화를 하면 컴파일러의 방어적인 구현이 작동하지 않고, 성능이 좋아짐
    - 이 경우 정의되지 않은 결과
  - 컴파일러가 이 키워드를 무시해 방어적인 구현이 작동하면, 성능은 저하
    - 정의되지 않은 결과를 막을 수 있음
