### base32, base64

![img_14.png](images/img_14.png)

- 1바이트(8비트)를 만들기 어렵기 때문에 잘 사용하지 않음

![img_15.png](images/img_15.png)

- 허용되지 않는 문자(예를 들어 '/')를 URL에 포함할 때
    - 문자를 비트 패턴으로 변환하는 과정이 필요함
    - 비트 패턴으로 '01010111....'로 2진법으로 표현하지 않고 다른 진법으로 표현하면 가독성을 올릴 수 있음
    - 진법을 선택할 때 1글자를 표현할 때 많은 비트를 사용하는 진법은 URL 길이를 줄일 수 있음
        - 직관적으로 생각하면 8진수 2글자(2자리)가 64진수 1글자와 동일함

![img_16.png](images/img_16.png)

- HTML에서 PNG(이미지)를 파일(문서)에 포함할 때 경로를 명시하면 브라우저에서 경로의 이미지 파일을 읽어와서 보여주게 됨
    - 하드의 파일을 읽어오거나
    - 서버에 요청을 보내 이미지를 받아오거나

- 경로를 명시하는 대신 embed(HTML 파일 속에 포함하는 방식)을 사용하면 이미지 파일의 비트 패턴을 문서에 명시함
    - 위의 예시에서는 '00010101000...' 와 같이 2진수로 표현하는 대신 base64로 표현
    - 위의 예시에서는 이미지의 크기가 21KB라서 21000 * 8 == 168000 비트로 2진수로 표현하면 너무 길다
    - base64를 사용하면 이 길이를 6배 줄일 수 있음

- 참고로 'index.html' + 'logo.png' 용량을 합치면 22KB인데, 'index_embed.html'의 용량은 28KB
    - 이미지 파일을 임베드 했을 때 용량이 더 커짐
    - 이유는 base64(6비트)라서 1바이트(8비트)에 나눠 떨어지게 저장하지 못하기 때문
    - 'logo.png'는 비트 패턴이라 1비트 단위로 1바이트에 나눠 떨어지게 저장하기 때문에 효율적으로 저장

## 엔디언

- 최소 저장단위를 `초과하는` 데이터를 저장할 때 저장 순서를 정의
    - 여기서 '최소 저장단위'는 1바이트
        - 메모리에 1바이트 단위로 읽거나 쓸 수 있음
        - 현재 대부분 컴퓨터에서 적용됨
    - 엔디언은 플랫폼에 따라 규칙이 다름

![img_114.png](images/img_114.png)

- 리틀 엔디언:
    - 데이터가 끝나는 마지막 단위가 가장 작은 메모리 주소에 위치
    - 요즘 CPU-메모리 구조에 사용
    - 예를들어 리틀 엔디언으로 설계된 x86 CPU 'store word 0x1234'라는 명령을 내리면, 메모리에 [0x34], [0x12] 순서로 저장됨
        - 이 CPU의 워드(논리적 연산 단위)는 2바이트, 메모리에 저장되는 최소 단위는 1바이트

<br/>

- 빅 엔디언:
    - 데이터가 끝나는 마지막 단위가 가장 큰 메모리 주소에 위치
    - 네트워크 스트림에 사용
        - 전송할 때는 사람이 읽기 쉽게 순서대로 보낸다고 생각하면 됨
        - '0x1234'를 전송하고 싶으면 총 2바이트의 바이트 스트림에 [0x12],[0x34]를 순서대로 전송

### 인코딩 스펙과 용어 정리

- 코드 포인트 (Code Point):
    - 유니코드 표에서 “U+0041” 같은, 추상적인 문자 번호

- 코드 유닛 (Code Unit):
    - 코드 포인트를 스펙이 정한 크기의 비트 블록(최소의 논리적 단위)로 나눈 것
    - UTF-8은 1바이트
    - UTF-16은 2바이트
    - UTF-32는 4바이트
    - 코드 포인트는 인코딩 스펙에 따라 하나 이상의 코드 유닛의 조합으로 구성될 수 있음

- 바이트:
    - 메모리나 파일/네트워크에서 실제로 전송/저장하는 최소 단위 (현대 플랫폼에선 8비트)
    - 코드 유닛의 조합(하나의 문자)는 바이트 스트림으로 풀어서 전달

- UTF-8은 왜 엔디언 문제가 없을까?
    - 코드 유닛이 1바이트이기 때문
    - 'byte-oriented' 라고 부름
    - 즉 최소 단위인 코드 유닛이 1바이트이기 때문에 전송/저장 시 바이트 단위로 읽으면 코드 유닛은 플랫폼의 영향을 받지 않음

- 아니 그러면 UTF-8 인코딩에서 여러 코드 유닛을 사용하는 문자에서 코드 유닛의 순서는 어떻게 결정하나요?
    - 이 때문에 아래에서 살펴볼 '첫 번째 바이트의 비트 패턴'으로 이 문자가 몇 바이트짜리 글자인지 알 수 있도록 스펙을 정의했음
    - 코드 유닛의 순서는 엔디언과 헷갈리기 쉬운데 독립적인 개념으로 바이트 스트림 해석에 대한 개념
        - 예를 들어 UTF-8 인코딩 결과 4바이트 크기의 글자(알파라고 부르자)는 코드 유닛 4개로 전송되고 코드 유닛의 스트림에서 어떤 코드 유닛부터 읽어야 알파를 읽을 수 있는가?

<br/>

- 지금 예시처럼 코드 유닛이 2바이트 이상이면 엔디언 문제가 발생
    - UTF-16:
        - 디코더는 바이트 스트림에서 2바이트씩(코드 유닛) 꺼내어 문자를 구성하게 됨
        - 이 때 코드 유닛(2바이트) 내부에서 순서가 달라질 수 있는데, 이게 엔디언 문제
        - 위의 예시를 기준으로 어떤 플랫폼에서 부터 [0x4C],[0xF3]를 바이트 스트림으로 전달받았다고 하자
        - 기본적으로 디스크나 파일 시스템은 그저 메모리에서 읽어온 바이트를 그대로 전송함
        - 따라서 플랫폼의 시스템이 메모리에 처음에 저장할 때 어떤 순서로 저장했냐가 중요함
        - 엔디언 문제를 해결하기 위해서 플래그를 넣어 이제부터는 "LE" 또는 "BE"로 해석하라고 알려줌
    - UTF-32
        - 유사함

> '논리적 저장단위'라는 표현은 일반적인 공식 용어는 아니지만, 질문자분의 맥락에서 보자면 개념을 잘 이해하신 것으로 보입니다.
> 조금 더 명확히 설명하자면, 현재 우리가 사용하는 대부분의 데스크탑 CPU에서 메모리는 1바이트(8비트) 단위로 주소를 매깁니다. 이 1바이트가 바로 '최소 저장단위'입니다. 그런데 어떤 데이터가 2바이트 이상을 차지한다면, 예컨대 ushort나 UTF-16 코드 유닛처럼 2바이트짜리 데이터가 있다면, 이 데이터를 여러 바이트에 나눠 저장해야 하므로 그 저장 순서—즉, 엔디언(Endianness)의 영향을 받게 됩니다.
> UTF8은 1 바이트를 1~4개 저장, UTF16은 2 바이트를 1~2개 저장, UTF32는 4 바이트를 1개 저장합니다. 따라서 UTF8은 엔디안의 영향을 받지 않지만, UTF16과 UTF32는 엔디안의 영향을 받습니다.

### UTF-8의 장점 2

![img_115.png](images/img_115.png)

- CJK를 제외한 거의 모든 문자에 1바이트 또는 2바이트 사용

- 하지만 한국어는 3바이트가 필요함
    - UTF-8에서는 적혀있는 문자가 몇 바이트인지를 표기하는 비트를 사용함
    - 존재하는 비트 모드를 모두 문자를 표현하는데 사용하지 못하기 때문에 이런 낭비가 발생

<br/>
- 반면 UTF-16에서는 한국어를 2바이트로 표현 가능하다
    - 하지만 UTF-8은 CJK를 제외한 나머지 문자에서 바이트를 적게 사용하기 때문에

### UTF-8의 장점 3

![img_116.png](images/img_116.png)

- 문자가 몇 바이트인지를 표기하는 비트를 사용
- 첫 번째 바이트(그림에서 바이트1 열)을 보는 것 만으로도 이 문자가 몇 바이트로 인코딩 된지 알 수 있음
    - 1바이트 사용하면:
        - 0
    - 2바이트 사용하면:
        - 110
    - 3바이트 사용하면:
        - 1110
    - 4바이트 사용하면:
        - 11110
- 나머지 바이트는 모두 '10'으로 시작하함
- '10' 과 몇 바이트로 인코딩 된지 알게 해주는 바이트를 제외한 'x'로 표기된 비트패턴이 진짜 문자를 표현하는 비트 패턴임
- 이 규칙 때문에 바이트 스트림을 순서대로 읽었을 때 "1110xxxx" 바이트를 읽었다면, 추가적으로 2개 바이트를 더 읽어서 3바이트 짜리 문자를 구성할 수 있음
    - 코드 유닛 3개를 조합

<br/>

- 4바이트를 사용하는 경우 'x'의 개수를 세보면 21개
    - 넉넉하다!

<br/>

- 단점:
    - 바이트를 낭비

### UTF-8 예시

![img_117.png](images/img_117.png)

- 'A'는 아스키 코드와 대응되는 '65'
    - 16진수로 '41'
    - 비트패턴으로 '100_0001'
    - 7비트만 사용하면 됨
- UTF-8로 인코딩하면 1바이트 사용
    - '0xxx_xxxx'에 7비트 채우면
    - '0100_0001'

<br/>

- '걁'은 유니코드 포인트로 보면 '0xAC41'
    - 총 16비트 필요하다
        - 사실은 2바이트로 표현함
    - 하지만 2바이트에서 몇 바이트인지를 나타내는 기본 비트를 제외하면 공간이 부족함
        - 따라서 3바이트 사용
- UTF-8로 인코딩하면 3바이트 사용
    - '1110xxxx_10xxxxxx_10xxxxxx'
        - 3바이트 사용하면 16비트까지 넣을 수 있음
    - 'A' == '1010'
    - 'C' == '1100'
    - '4' == '0100'
    - '1' == '0001'
    - 'x' 공간에 왼쪽에서 부터 채우면 됨

<br/>

- 3바이트를 사용하더라도 1바이트 단위로 인코딩하기 때문에 '0xEA', '0xB1', '0x81' 엔디언의 영향을 받지 않음
    - 코드 유닛이 1바이트
